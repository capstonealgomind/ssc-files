SSCEVS — Student Council Election Voting System

OVERVIEW
--------

SSCEVS (Student Council Election Voting System) is a web application for
managing student council elections at an educational institution. It covers the
full election lifecycle: voter registration with ID verification, admin review,
secure online voting, live monitoring, ballot receipts, results, and support.

The system is built for BCC-style (Baao Community College / similar Philippine
college) student IDs, with OCR tuned for common ID layouts and student ID
formats (e.g. 23-1-0516).

USER ROLES
----------

VOTER
  Students who register, get verified, and cast ballots.

ADMIN
  Election administrators with full management access. Admin emails use the
  domain @sscevs.admin.com (e.g. admin@sscevs.admin.com).

  Role is determined at registration/login from the email domain:
  - Emails ending in @sscevs.admin.com → admin
  - All other emails → voter


CORE FEATURES
-------------

FOR VOTERS
  - Multi-step registration (profile → ID scan → OTP → admin/auto verification)
  - Elections page with candidate cards per position
  - One ballot per election (one vote per position)
  - Ballot receipt with PDF download (signed URL)
  - My Votes, Results, Announcements
  - Help & Support tickets with live chat (after admin approval)
  - AI-powered FAQ chat (SSCEVS-scoped)
  - Profile photo and password management
  - Public registration status check (no login required)

FOR ADMINS
  - Election CRUD (draft → scheduled → active → closed)
  - Candidate management (photo, platform, position, department, partylist)
  - Voter list with OCR data, fraud score, and verify/reject actions
  - Live election monitoring (turnout, standings, analytics)
  - Support ticket approval and chat
  - Settings: departments, courses, year levels, positions, partylists
  - Admin account management
  - Reports page

ELECTION STATUSES
-----------------

  draft      → Not yet published
  scheduled  → Published, voting may not have started
  active     → Election is live
  closed     → Election finished

Voting is open when status is active or scheduled AND the current time is
within voting_starts_at and voting_ends_at.


REGISTRATION FLOW
-----------------

Step 1: Register form
  → Name, email, student ID, department, course, year level, password
  → Data stored in session (no user created yet)

Step 2: ID Scan
  → Capture/upload school ID photo (client-side quality check: good/warn/blurry)
  → User account created with voter ID (VID-YYYY-XXXXX)
  → Verification email + OTP sent (queued)

Step 3: OTP verification
  → Voter enters 6-digit OTP (or clicks email verification link)
  → Email marked verified; OCR job queued

Step 4: Background OCR + fraud scoring
  → ID image processed (OCR.Space → Tesseract fallback)
  → Fraud score calculated from OCR match, duplicates, device fingerprint, etc.

Step 5: Auto or manual verification
  → Score >= 80: auto-approved (is_verified = true)
  → Score < 80: pending admin review on /voters page
  → Admin can Approve, Reject, or Re-run OCR

FRAUD DETECTION SCORING
-----------------------

FraudDetectionService assigns points based on:

Signal                                          | Points
------------------------------------------------|--------
OCR name matches typed name (>=70%)             | +20
Student ID matches OCR                          | +20
Course matches OCR (>=60%)                      | +10
Email verified (OTP)                            | +20
Email local part resembles name                 | +20
Good image quality                              | +10
Blurry image                                    | -20
Duplicate email                                 | -50
Duplicate student ID                            | -100
Multiple registrations from same device (24h)   | -30
Too many OTP failures (1h)                      | -20

Score labels:
  >= 80  → High confidence
  >= 50  → Moderate – review recommended
  >= 20  → Low – manual review required
  < 20   → Flagged – likely fraudulent


VOTING FLOW
-----------

1. Voter must be verified (is_verified = true).
2. Election must be in open voting phase.
3. Voter selects one candidate per position.
4. Server validates: all positions filled, no duplicates, candidates belong
   to election.
5. Votes and a BallotReceipt are created in a database transaction.
6. Voter is redirected to the receipt page; PDF available via signed URL
   (1-hour expiry).

Votes cannot be changed after submission.

VOTING PHASE DECISION (per election)
------------------------------------

  Election status in [active, scheduled]
           AND
  now >= voting_starts_at  AND  now <= voting_ends_at
           AND
  status NOT in [draft, closed]
           │
           ├──▶ voting_phase = "open"     → voter can cast ballot
           ├──▶ now < voting_starts_at    → voting_phase = "upcoming"
           └──▶ now > voting_ends_at       → voting_phase = "closed"


OCR PIPELINE
------------

OcrService tries OCR.Space first, then local Tesseract as fallback. It extracts:

  - Name       — multiple patterns for Philippine school IDs
  - Student ID — e.g. YY-1-XXXX (BCC format)
  - Course     — degree/program text

Processing runs asynchronously via ProcessOcrVerification job (3 retries with
backoff: 60s, 300s, 900s).

  Laravel Echo + Pusher broadcasts SupportMessageSent when messages are posted

FAQ ASSISTANT
-------------

Powered by Groq (FaqChatService). Answers only SSCEVS-related questions;
off-topic queries get a standard redirect to Help & Support.

Rate-limited to 15 requests per minute.

Requires GROQ_API_KEY in .env.


INSTALLATION & SETUP
--------------------

Prerequisites:
  - PHP 8.2+
  - Composer
  - Node.js 18+
  - MySQL
  - (Optional) Tesseract OCR for local fallback
  - (Optional) Pusher account for real-time chat

Quick setup:
  composer setup

  This runs: composer install, copies .env, generates app key, migrates,
  npm install, and npm run build.

Manual setup:
  composer install
  cp .env.example .env
  php artisan key:generate

  # Configure DB_* in .env, then:
  php artisan migrate
  php artisan db:seed
  php artisan storage:link

  npm install
  npm run build

Development:
  composer dev

  Starts concurrently:
    - php artisan serve
    - php artisan queue:listen
    - npm run dev (Vite HMR)

  The queue worker is REQUIRED for OCR processing and verification emails.

TYPICAL ADMIN WORKFLOW
----------------------

1. Configure Settings (departments, courses, year levels, positions, partylists).
2. Create an Election and set voting window.
3. Add Candidates for each position.
4. Set election status to Active when voting should open.
5. Review pending Voters (check fraud score and OCR data).
6. Monitor turnout on Monitoring during voting.
7. Close election and publish Results.


TYPICAL VOTER WORKFLOW
----------------------

1. Register with student details.
2. Scan/upload school ID.
3. Verify OTP from email.
4. Wait for account verification (auto or admin).
5. Cast vote during the Elections window.
6. Download ballot receipt as proof.
7. View Results after the voting period ends.
